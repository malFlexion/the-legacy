# Chapter 10 - Creating a Coding Copilot: Integrating an LLM Service into VS Code with RAG

## Overview
This chapter builds a VS Code coding copilot extension backed by a local LLM (DeciCoder 1B) enhanced with a RAG system using Milvus as the VectorDB. It covers model serving via FastAPI, data ingestion into a vector database, building the VS Code extension in JavaScript, and lessons learned about when RAG helps versus hurts.

## Key Concepts

### Model Selection: DeciCoder
- **DeciCoder**: Commercial open source model, only 1B parameters, trained on Python, Java, and JavaScript from the Stack dataset
- Small context window of 2,048 tokens -- a constraint when using RAG since code samples are verbose
- Not instruction-tuned; trained to beat the HumanEval benchmark (given function name + docstring, generate the implementation)
- Its tiny size makes it a candidate for embedding directly in a VS Code extension via GGUF compilation (left as an exercise)

### Serving the Model via FastAPI
- Model served as a local REST API using FastAPI + uvicorn
- **Stop tokens** (`def`, `class`, `Instruction`, `Output`) implemented via a custom `StoppingCriteria` class to limit generation to one function at a time
- Standard pattern: tokenize input, call `model.generate()` with stopping criteria, decode only the new tokens (strip input prefix)

### VectorDB Setup with Milvus
- **Milvus**: Open source VectorDB; runs locally via Docker standalone, scales to cloud clusters, has a GPU-enabled version
- A **collection** in Milvus is like a table; schema includes ID, embeddings (FLOAT_VECTOR), and metadata (JSON)
- **Index type IVF_FLAT**: Splits embedding space into `nlist` clusters; search compares query against cluster centers first, then within the closest clusters (`nprobe` parameter controls how many clusters to search)
- **Metric**: L2 (Euclidean distance) for similarity measurement

### Data Pipeline and Embeddings
- Uses the Alpaca Python subset dataset (18.6K Python coding challenges with instruction + generated code)
- **Embedding model**: `krlvi/sentence-t5-base-nlpl-code_search_net` -- specifically trained on CodeSearchNet for meaningful code embeddings
- Text chunked with `RecursiveCharacterTextSplitter` (chunk_size=400, chunk_overlap=20) using DeciCoder's tokenizer for length measurement
- Batch upload with UUID-based IDs; embeddings generated by SentenceTransformer

### RAG Integration
- Updated API encodes user prompt with the same embedding model, searches Milvus for top 5 nearest neighbors
- Results formatted as few-shot prompt examples: `Instruction: ... Output: ...`
- Token counting ensures examples fit within the 2,048 token context window; excess examples are dropped
- Full prompt structure: system instruction + filtered examples + user input

### Building the VS Code Extension
- Scaffolded with `yo code` generator; two key files: `package.json` (manifest) and `extension.js` (logic)
- **Activation event**: `onCommand:editor.action.inlineSuggest.trigger` -- fires on manual trigger, not every keystroke
- **Custom keybinding**: Alt+S mapped to trigger inline suggestions
- **Inline completion provider**: Returns ghost text at cursor position; user accepts with Tab
- Provider grabs highlighted text, sends async fetch request to local API, returns response as inline suggestion
- Handles backward selection by re-highlighting to place cursor at end before retriggering

### Lessons Learned
- RAG with the Alpaca dataset actually degraded results for DeciCoder because the model wasn't instruction-tuned; it got confused by plain-English few-shot examples
- Without RAG, DeciCoder performed better on code completion tasks because that matches its training (code-in, code-out)
- The mismatch between how users interact (sending half-written code) and RAG examples (plain English task descriptions) also contributed to degradation
- Prompt design must account for the model's training methodology -- blindly adding context can hurt

## Key Takeaways
- A small, specialized model (1B params) can outperform expectations when well-matched to its task
- RAG is not always beneficial -- if the model wasn't trained for the input format your RAG system produces, it may hurt performance
- Milvus with IVF_FLAT indexing provides fast, scalable vector search suitable for local development and production
- Building a VS Code extension is achievable with basic JavaScript; the inline completion provider API handles most of the UX complexity
- Always consider how users will actually interact with the system when designing prompts and RAG pipelines
